# 更新日志

[English](CHANGELOG.en.md)

这里只记录用户能感受到的变化；实现细节见 Git 提交或 Pull Request。

## 3.4.2 - 2026-07-19

**所有用户文档都先讲结果，也更短了。**

- 以前，早期 changelog 仍像工程记录，README 和平台指南也重复解释内部流程；现在，44 个历史版本都先说用户能感受到什么，必要时再说明 Before → After、对应 subskill、升级方式和重要限制。
- README 现在先回答“Teamwork 能帮我做什么、怎样最快开始”；Codex、Cursor、Claude Code 指南分别保留各平台真正需要的安装、手动步骤、日常用法和排错信息。Marketplace 描述也改成了简短的用户收益说明。
- 后续公开文档统一遵守同一规则：简洁第一，先讲用户能完成什么；内部架构和维护过程只在会改变操作、边界或预期时出现。
- Codex Marketplace 不再推荐固定在某个版本 tag；以前，`upgrade` 只能刷新同一个 tag，无法前进到新版本；现在，移除旧 `teamwork` Marketplace 后重新添加未固定版本的 `JinPLu/Teamwork`，后续更新即可正常前进。这个修复由 `teamwork-update` 负责。
- 升级方式：已有 3.4.1 用户若只需要原有能力，无需升级；要修复后续 Marketplace 更新，按 README 的更新命令重新添加一次未固定版本的 Marketplace。

限制：文档更短不代表 Teamwork 能保证每次自然语言请求都触发指定 skill；选择仍由宿主与模型决定。重新添加 Marketplace 仍需开启新任务并运行 `$teamwork-update`。

## 3.4.1 - 2026-07-19

**更新日志现在先告诉你“用起来哪里变了”。**

- 以前，重要变化常被写成安装、验证和维护报告；现在，每项变化先讲结果，需要时再用 Before → After 解释，并在有帮助时说明来自哪个 subskill 或工作流。
- 3.4.0 到 3.1.0 的最近几条更新已按这个标准重写，后续版本也会沿用；3.4.2 随后把同样的写法扩展到了全部历史版本和主要用户指南。
- 升级方式：无需升级。3.4.1 只改文档与发布写法，Teamwork 的运行行为没有变化。

限制：subskill 名称只说明变化来源；实际是否触发仍由宿主与模型共同决定。

## 3.4.0 - 2026-07-18

**Codex 现在可以从 Marketplace 一键安装 Teamwork。**

- 以前，Codex 用户要 checkout 仓库并运行安装器；现在，安装 `teamwork-skill@teamwork` 后，新任务里的 `$teamwork-update` 会展示完整设置，并在你同意后启用 agents、路由、规则和可选通知。主要由 `teamwork-update` 负责。
- Marketplace 安装不会悄悄改配置。它先让 skills 从插件缓存可用，完整配置另行确认；只清理确认属于 Teamwork 的旧 skills，也不再创建 `~/.agents/skills` 副本。
- `grill-me` 更准确地保存值得继续的讨论；范围和权限已清楚的普通任务仍由 `teamwork-execute` 直接推进，不把 plan 当成执行前提。
- 升级方式：运行 `codex plugin marketplace add JinPLu/Teamwork@v3.4.0`，再运行 `codex plugin add teamwork-skill@teamwork`，开启新任务并调用 `$teamwork-update`。checkout 用户在 3.4.x 仍可运行 `./install.sh all` 和 `./scripts/check-update.sh --readiness`；Cursor 与 Claude Code 不变。

限制：完整启用后要重启 Codex；通知还需在 `/hooks` 中只信任 Teamwork 的 `Stop` 和 `PermissionRequest`。插件不会静默改配置或信任 hook；损坏的讨论历史不会自动修复。安装检查也不能保证每次请求都会触发某项 skill，或证明三个平台行为完全一致。

## 3.3.0 - 2026-07-16

**不乱 test！优先完成任务再复查！**

- 以前，简单修改也可能膨胀成调研、计划、测试、review 和报告循环；现在，`teamwork-execute` 在授权清楚时先走最短真实路径，只检查当前 blocker、实际改动或明确的高风险边界，结果出现就停。
- `using-teamwork` 更容易从自然语言识别“先问清楚”“查原因”“按现有方案继续”等意图，并交给 `grill-me`、research、debug 或 execute；简单请求仍保持简单。
- 回答先给结论。只有你明确要求先讨论、项目已经初始化，而且确实有值得下次继续的结论或问题时，`grill-me` 才留记录；普通 plan 不会创建讨论记录。全局更新归 `teamwork-update`，项目说明与上下文归 `teamwork-init`，两者不再混用。
- 升级方式：运行 `./install.sh all`，再运行 `./scripts/check-update.sh --readiness`。已有项目如需完整上下文与讨论连续性，再运行 `./install.sh --project-root "<项目路径>" init-project`。

限制：路由由宿主与模型共同决定，不能保证每回合都走最短路径；发布、迁移、权限、安全和破坏性操作仍需明确边界。Cursor User Rules 仍要按提示手动粘贴。

## 3.2.0 - 2026-07-16

**讨论更像自然交流，回来也更容易接着聊。**

- `using-teamwork` 会把结论、依据、通俗解释和真正影响决定的边界连起来说，不再让版本号、内部术语或重复保留意见打断回答；观察与推断也会分开。
- `grill-me` 会记住已经谈定的结论和下一项待比较、测量或决定的问题。以前可能漏掉继续点，或先播报保存过程；现在只告诉你记住了什么、接下来聊什么。
- 升级方式：当时的指引是运行 `./install.sh all --profile gpt56-role`，再运行 `./scripts/check-update.sh --readiness`。Cursor User Rules 仍需手动粘贴；已有项目无需复制 Teamwork 包。

限制：这不是论文写作功能，也没有新增学术写作依赖。有限的离线与 Codex 场景检查不能保证每次回复同样自然，也不能证明 Cursor 与 Claude Code 行为一致。

## 3.1.0 - 2026-07-15

**长讨论中断后，回来还能从未决问题继续。**

- `grill-me` 只在确实需要跨会话继续时保存一份紧凑摘要，记住目标、已定选择、未决问题、关键依据和继续点；短讨论不创建文件，恢复时也不重问已定事项。
- 普通回答由 `using-teamwork` 先说结论和真正影响决定的事实，不再让过程与版本信息挡住重点。
- `teamwork-init` 能在项目锁内恢复中断的初始化，并拒绝未知文件、损坏日志和被复用的临时资源，不从不可信的半成品继续。
- 升级方式：运行 `./install.sh all` 和 `./scripts/check-update.sh --readiness`。已有项目无需复制包；需要刷新项目上下文时，运行 `./install.sh --project-root "<项目路径>" init-project`。

限制：只实测了一个隔离 Codex 场景；新会话恢复不等于宿主真实 compaction，也不证明 Cursor 和 Claude Code 行为一致。

## 3.0.0 - 2026-07-15

**回复更直接，项目里也不再复制一整套 Teamwork。**

- 以前，普通回复可能先讲阶段、版本和内部标签；现在先说结论、重要原因和下一步，只在影响决定或你明确要求时补实现细节与不确定性。
- 长讨论、交接或重要分支在有写权限时可保存路线图和文字回放；普通对话不会自动成为工程记录，没有写权限时也只能提供候选检查点。
- 项目内 Teamwork 包副本被移除，`project`、`project-codex-agents` 和 `check-update --project` 不再可用。`init-project` 只写项目说明、memory 和 CodeGraph context，并刷新全局 Teamwork。
- 升级方式：运行 `./install.sh all` 和 `./scripts/check-update.sh --readiness`；需要项目 context 时运行 `./install.sh --project-root "<项目路径>" init-project`。清理旧副本时只能删除确认由 Teamwork 生成的内容，不要整目录删除 `.agents`、`.codex`、`.cursor` 或 `.claude`。

限制：本版只有离线规则检查、安装模拟和整包验证，没有付费或真实模型对比，不能据此断言回答质量提升或三个平台行为一致。

## 2.20.0 - 2026-07-14

**改动优先复用现有路径，减少无必要的包装和备用流程。**

- 目标是减少重复概念，不是追求最少代码；需要时仍会跨文件修改、保持清晰模块并完成测试。过期、未接受或不再适用的 Teamwork 记录也不会继续引导当前任务。

限制：规则和安装检查不能保证每次回答都自动采用最简实现，也不代表三个平台已经完成全部运行时验证。

## 2.19.0 - 2026-07-13

**重要完成与权限请求提醒默认可用，而且只信任 Teamwork 自己的 hook。**

- `install.sh all` 和 `init-project` 默认安装 Codex/Claude Code 的主任务完成音与权限请求音；直接安装单个平台仍需显式开启，`--no-notifications` 可退出且只移除 Teamwork 处理器。
- `check-update.sh` 会区分 hook 已信任、待审核和无法核验；待审核时提示使用 `/hooks`，不会仅凭配置存在就称其可用。
- `teamwork-init` 和 `teamwork-update` 只核对 Teamwork 的准确命令，并分别信任 `Stop` 与 `PermissionRequest`，不会信任其他 hook。

限制：提示音只在 macOS 实测。

## 2.18.0 - 2026-07-13

**Teamwork 会先查证，只在确实需要你决定时提问。**

- 缺少必须由你提供的输入、观察，或有会显著改变结果的决定时才提问；一个分支等待确认时，独立的只读调查仍可继续。子任务只提交候选问题，由主任务合并后统一询问。
- 2.17 的版本化任务契约被轻量工作事实取代。复查只让违反已接受条件、破坏边界、引入回归或缺证据的问题阻断完成；Goal 也只在你明确要求或接受提议后启动。
- 发布核对会同时检查版本、双语更新日志、安装内容、远端标签和 GitHub Release；缺少标签或 Release 时只能称为 `release-ready`。

限制：离线与有限 Codex 实测不能证明所有阶段、自然语言问题去重或 Cursor/Claude Code 的完整运行时行为。

## 2.17.0 - 2026-07-13

**重要方向会提前对齐，修复与复查更快收敛。**

- 非简单方案会先查仓库与配置，一次只问一个必须由你决定的问题并给出推荐，最终方案前提供简短决定摘要。
- 需要独立验收时最多一次完整审查；修复后同一审查者只增量复查原问题与新回归。只有违反已接受条件、破坏边界、引入回归或缺证据才是 blocker。
- 已知原因直接修，未知原因先诊断；只有你接受范围变化才重做计划。进度只报告新决定、阻塞、验证和完成状态。
- 当时使用版本化任务契约保护非简单任务，2.18 后改为轻量工作事实；固定“零到三个问题”和“进行中/已结束”仪式也从本版起取消。

限制：方案与复查能力仍在，Codex 角色模型档位未降低；本版减少重复流程，但不承诺消除 GPT-5.6 Sol/high 的单次延迟。

## 2.16.0 - 2026-07-13

**`grill-me` 首次成为独立 skill，只问真正影响结果的问题。**

- 当时每轮问零到三个必须由你决定的问题，可逆的语言、文件数、命名和内部布局不会用来凑数；2.17 后取消固定数量。
- 多轮讨论当时保留“进行中/已结束”状态，问完问题也不获得实施权限；2.17 取消这套状态，2.18 又用轻量工作事实替代版本化任务契约。
- Codex 开始按职责使用各自安装的模型与推理强度，而非一律继承主任务；并发上限为 9 个线程，即主任务加最多 8 个子任务。当时的项目级安装不会修改用户级配置，并保留明确的退出选项。

限制：需重启 Codex 并开启新任务才读取新配置；只在 Codex 0.144.0 做过有限实测，跨平台 `grill-me` 仍只有静态与离线验证。

- 可选提示音只提醒主任务，后台任务保持静音；另有只读工具检查 agent 配置和异常长任务，不输出对话正文。

限制：提示音只在 macOS 验证；Claude Code 当时未实测，Cursor 不支持。
