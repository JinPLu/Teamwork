# 更新日志

[English](CHANGELOG.en.md)

这里只记录用户能感受到的变化；实现细节见 Git 提交或 Pull Request。

## 3.1.0 - 2026-07-15

这次更新让长讨论真正可恢复，也让普通回答更接近用户需要的结论。

- 讨论连续性：之前持久化依赖模糊判断和多处手写同步；现在只有出现明确的连续性信号时，才保存一份紧凑的五段式摘要：目标、已定选择、未决问题、关键证据和继续点。新会话只恢复未决事项，不重问已定选择；问题解决后一次关闭。短讨论仍不创建文件。
- 面向用户的回答：之前工程过程、版本号、自创标签和重复的保留意见可能遮住答案；现在先给结论，再给影响决定的事实、机制和下一步。解释能力或限制时，有用的 skill 名称仍可保留。
- 项目初始化更可靠：初始化在进程中断后可在项目锁内恢复，并拒绝未知文件、损坏日志和临时资源复用，避免从半完成状态继续。
- 升级方式：在 Teamwork 仓库运行 `./install.sh all`，再运行 `./scripts/check-update.sh --readiness`。已有项目不需要复制新的 Teamwork 包；需要刷新项目上下文时，运行 `./install.sh --project-root "<项目路径>" init-project`。

限制：真实行为验证覆盖一个隔离的 Codex 场景；新会话恢复不等同于宿主实际 compaction，也不证明 Cursor 和 Claude Code 的运行时一致性。

## 3.0.0 - 2026-07-15

本版让普通回复更直接，也简化了项目安装方式。

- 回复语言风格优化：普通回复不再以内部流程开场，过去可能先出现版本号、阶段、自创标签或重复的保留意见；现在先说明结论、为什么重要以及下一步。只有在你要求，或细节会影响决定与行动时，才补充内部实现、版本和不确定性。
- 长讨论连续性：长讨论、交接或重要分支在具备写权限时，可以留下可恢复的路线图和文字回放；没有写权限时只提供候选检查点，无法保证之后能恢复。普通对话不会因此自动变成工程记录。规则变化还必须说明由谁负责、会怎样影响用户以及如何验证。
- 项目安装变更：项目内的 Teamwork 包副本已取消，`project`、`project-codex-agents` 和 `check-update --project` 已不可用。`init-project` 现在刷新全局 Teamwork，并且只向项目写入说明、memory 和 CodeGraph context，不会复制 skills 或 agents。
- 升级方式：在 Teamwork 仓库运行 `./install.sh all`，再运行 `./scripts/check-update.sh --readiness`。需要项目 context 时，运行 `./install.sh --project-root "<项目路径>" init-project`。清理旧副本时，只删除已经确认由 Teamwork 生成的内容；不要整目录删除 `.agents`、`.codex`、`.cursor` 或 `.claude`，这些目录可能还有你自己的文件。

验证范围：本版本只经过离线规则检查、安装模拟和完整包验证；没有运行付费或真实模型对比，因此不能据此断言回答质量已经提升，也不能断言 Codex、Cursor 和 Claude Code 的实际运行行为完全一致。

## 2.22.0 - 2026-07-15

规则维护更集中：共同边界集中维护，各阶段只保留自己的职责和完成条件，减少全局规则、各阶段 skill、agent 模板和项目初始化之间的重复，简单任务仍可直接处理。

- 发布隐私保护：真实用户目录、会话标识、私有网络地址、疑似凭证内容，以及被强制跟踪的私有运行产物不会进入开源包；明确公开的传输配置和合成测试值仍可保留。
- 项目初始化更稳妥：默认生成精简、可迁移的 Teamwork 说明和先列标题的索引；已有的完整旧索引仍兼容，混合但不完整的格式会被拒绝，重复运行也不会改写现有内容。
- 维护者候选比较：两个匿名候选由两名独立审查者按固定案例和硬性条件评分；原始运行记录保持私有，只有实质结果完全打平时，才有限参考规则体积。普通用户无需操作。
- 2.22.0 当时的升级方式：在 Teamwork 仓库运行 `./scripts/check-update.sh --readiness --project "$PWD"`，执行输出中的 `NEXT`，再重复检查，直到出现 `INSTALL_READY=yes`。Cursor User Rules 当时仍需按提示手动粘贴。3.0.0 后来移除了这些项目内更新路线；当前安装请按上方 3.0.0 的说明操作。

验证范围：本版本经过离线规则检查、安装模拟、隐私扫描、成对评测流程和完整包验证；没有运行付费模型对比，因此不能据此断言真实回答质量提高，或 Codex、Cursor 和 Claude Code 的运行时行为等价。

## 2.21.1 - 2026-07-15

维护入口保持兼容：安装、离线评测和整包验证仍使用相同入口和输出，改动位置更容易找到，普通任务不会因此增加步骤。

- 安装后抽查：维护者可以在私有临时环境中选择抽查一次真实安装结果，只留下脱敏摘要，不保留原始对话内容。
- 2.21.1 当时的升级方式：在 Teamwork 仓库运行 `./scripts/check-update.sh --readiness --project "$PWD"`，执行输出中的 `NEXT`，再重复检查，直到出现 `INSTALL_READY=yes`。当时无需新增用户配置，Cursor User Rules 仍需按提示手动粘贴。3.0.0 后来移除了项目内更新路线；当前安装请按上方 3.0.0 的说明操作。

版本定位：这是兼容性维护补丁，不代表普通模型回答优于 2.21.0。一次有限抽查不能证明整体可靠性、自动启用行为或三个平台的运行时等价性。

## 2.21.0 - 2026-07-15

长讨论连续性：经过压缩、暂停或交接后，符合条件的长讨论可以用路线图和文字回放保留已接受、已拒绝和待决定的方向，更容易恢复上下文。

- 短讨论仍保持轻量：普通短讨论不创建文件；保存内容只记录关键路线，不保留逐轮对话，也不会因此获得执行权限。没有写权限时只提供候选摘要，无法保证后续连续性。
- 初始化内容保护：项目初始化会根据证据处理保留、合并、迁移、删除、新建和未解决项，并保护已有的人类文档、项目追踪器和自定义内容。
- 更新与发布分权：更新已安装内容不会隐含公开发布权限；只有维护者明确提出发布请求，才会改动版本、更新日志、标签或 GitHub Release。
- 2.21.0 当时的升级方式：在 Teamwork 仓库运行 `./scripts/check-update.sh --readiness --project "$PWD"`，执行输出中的 `NEXT`，再重复检查，直到出现 `INSTALL_READY=yes`。Cursor User Rules 当时仍需按提示手动粘贴。3.0.0 后来移除了项目内更新路线；当前安装请按上方 3.0.0 的说明操作。

能力限制：有限检查支持这项流程，但不能保证每次真实讨论都能完整恢复，也不能证明三个平台的运行时行为完全一致。

## 2.20.0 - 2026-07-14

实现路径更精简：局部改动优先复用已有负责路径和平台能力，只有确有必要时才增加抽象或依赖，减少包装层、配置模式、备用流程和重复模块带来的维护负担。

- 代码质量边界：这不是要求“代码越少越好”；任务需要时仍应跨文件修改、保持清晰模块并完成测试，目标是减少重复概念，而不是牺牲质量压缩行数。
- 当时的三平台项目安装：本版本曾为 Codex、Cursor 和 Claude Code 准备全部十个 Teamwork skills，并保留其他本地内容；该项目内副本路线后来已在 3.0.0 删除。
- 平台级更新诊断：在当时仍支持项目副本时，检查结果会分别说明 Codex、Cursor 或 Claude Code 的项目 skills 是缺失、过期还是内容不同，而不是只给一个笼统状态。
- 旧计划隔离：已过期、未接受或已不适用的 Teamwork 记录不再引导当前任务。
- 当时的升级方式：在 Teamwork 仓库运行 `./scripts/check-update.sh --readiness`；需要同步项目时加 `--project <项目路径>`。执行输出中的 `NEXT` 后，重新运行同一检查，直到出现 `INSTALL_READY=yes`。这是 2.20.0 的历史指引；3.0.0 已移除 `--project` 路线。

验证范围：这些变化由工作流约束和安装检查覆盖；它们不能保证每次模型回答都会自动达到最简，也不代表已经验证三个平台的全部运行时行为。

## 2.19.0 - 2026-07-13

关键提醒默认启用：`install.sh all` 和 `init-project` 默认安装 Codex/Claude Code 的主任务完成音与权限请求音；直接安装单个平台仍需显式选择启用。`--no-notifications` 可退出，并且只移除 Teamwork 自己的处理器。

- 提醒状态核验：`check-update.sh` 会通过 Codex 运行时区分已信任、待审核和无法核验；待审核时会提示使用 `/hooks`，不再把配置中写有 hook 当成已经可用。
- 权限范围收紧：`teamwork-init` 和 `teamwork-update` 只核对准确的 Teamwork 通知命令，并分别信任 `Stop` 与 `PermissionRequest`；不会信任全部 hook，也不会顺带信任其他 hook。
- 实测范围：macOS 提示音已实际验证。

## 2.18.0 - 2026-07-13

提问更克制：Teamwork 会先查看仓库、配置和已有证据；只有缺少必须由你提供的输入、观察，或需要你决定且会显著影响结果时才提问，普通澄清不必切换到 `grill-me`。

- 分支继续推进：一个分支等待确认时，独立的只读调查仍可继续，不会冻结整个任务。
- 问题统一整理：子任务只提交候选问题，由主任务重新核对并合并重复项后统一询问，不会各自打断你。
- 2.17 规则的后续变化：版本化任务契约被轻量工作事实替代，任务只保留继续工作所需的目标、范围、验收条件、权限、阻塞和停止条件。
- 复查与长期目标边界：只有违反已接受条件、破坏保护边界、引入回归或缺少必要证据的问题才会阻断完成；修复后最多针对原问题复查一次。长期目标只重试受影响部分，并且只有你明确请求或接受提议后才会进入目标模式。
- 维护者发布核对：版本、双语更新日志、安装内容、远端标签和 GitHub Release 会一起核对；缺少标签或 Release 时只能报告 `release-ready`，不能称为已发布。

验证范围：离线结果支持三平台规则和安装兼容性，有限 Codex 实测支持跨阶段提问与权限保持；尚不能据此证明所有阶段、自然问题去重或 Cursor/Claude Code 的完整运行时行为。

## 2.17.0 - 2026-07-13

重要决定提前对齐：非简单方案会先查看仓库和配置，再一次只问一个确实由你决定的问题并给出推荐；最终方案前会显示简短决定摘要，减少执行中途才发现方向错误。

- 复查次数收敛：需要独立验收的任务最多进行一次完整审查；修复后由同一审查者做一次增量复查，只看原问题、声明的修复和修复引入的回归。简单任务仍可直接自验。
- 完成边界明确：只有违反已接受条件、破坏保护边界、引入回归或缺少必要证据，才能成为 `BLOCKER`；偏好、顺手优化和范围外建议可以保留，但不能无限阻止验收。
- 已知问题直接修复：原因明确时直接实施，原因不明时先诊断，只有你接受范围变化才重新规划；验证失败也只重做受影响部分。
- 当时的目标保护规则：非简单、多阶段任务使用版本化任务契约，范围变化必须显式说明并升级版本，简单任务无需创建。2.18 后来用轻量工作事实取代了这套机制。
- 进度信息更安静：只报告新决定、阻塞、验证结果和完成状态，不再反复复述计划或宣布又一次复查。

版本边界：方案与复查能力没有被删除，Codex 的角色模型档位也没有降低。本版本针对重复流程、返工和收敛时间，不承诺消除 GPT-5.6 Sol/high 的单次响应延迟。相较 2.16，本版本取消了固定“零到三个问题”和“进行中/已结束”的状态仪式；2.18 后来又以轻量工作事实取代了本版本的任务契约。

## 2.16.0 - 2026-07-13

当时的 `grill-me` 交互：`grill-me` 成为独立 skill，只询问零到三个真正影响结果且必须由你决定的问题；语言、文件数量、命名和内部布局等可逆细节不会被拿来凑数。2.17 后来取消了固定问题数量。

- 当时的讨论状态：多轮讨论会保留明确的“进行中/已结束”状态；只有你要求停止、继续执行、委托判断或切换任务时，才按相应权限退出。问题问完也不会自动获得实施权限。2.17 后来取消了这套状态仪式；2.18 再以轻量工作事实取代了 2.17 的版本化任务契约。
- 子任务模型选择：Codex 会按各自的安装配置为不同职责选择模型和推理强度，不再一律继承主任务的推理强度。
- 并发上限：Codex 会话上限改为 9 个线程，即一个主线程加最多 8 个子任务；当时的项目级安装仍不修改用户全局配置，并保留明确的退出选项。
- 生效与验证范围：这项路由在 Codex 0.144.0 做过有限实测。需要重启 Codex，新任务才会读取新配置；跨平台 `grill-me` 行为仍只经过静态与离线验证。

## 2.15.0 - 2026-07-13

用户纠正立即生效：旧任务会随纠正失效，避免后台继续沿用过时方向，也减少重复协作。

- 完成声明更准确：局部完成或尚未验证的结果，不会再被说成整个任务已经完成。
- 提示音更克制：Codex 的可选提示音只提醒主任务，后台任务保持静音，避免并行工作频繁打扰。
- 只读诊断：新增工具可检查 agent 配置和异常长任务，而且不输出对话正文。
- 平台支持范围：提示音已在 macOS 验证；Claude Code 当时尚未实测，Cursor 不支持。

## 2.14.0 - 2026-07-11

Codex 模型配置更新：`performance-first` 改用 GPT-5.6 Terra/Sol，`cost-first` 改用 Luna/Terra/Sol；新增全部角色固定为 Sol 的 `gpt56-high` 和 `gpt56-xhigh`。

- 兼容配置名称：`gpt56-role`、`gpt55-high` 和 `gpt55-xhigh` 仍可作为兼容名称，但所有 Codex profile 都不再调用 GPT-5.5。
- Cursor 与 Claude Code 映射：Cursor 使用当时的 Composer 2.5、Sonnet 4.6 和 Opus 4.8 原生映射，其中 `cost-first` 使用非 Fast Composer 2.5；Claude Code 使用当时的 `haiku`/`sonnet`/`opus` 别名，Deep 档提升到 `max`。
- 安装结果一致：复制、链接、全局安装和当时的项目安装使用同一模型映射，不再因为入口不同而得到不同选择。

## 2.13.0 - 2026-07-10

信息充分时直接推进：不再仅为满足模板而强制列假设、画表格、创建长期记录或启动独立复查。

- 提问与验证按风险触发：只有真正影响结果、且必须由你决定的问题才会打断你；测试先行、方案对比、长期记忆和全新上下文复查不再一律强制。
- 上下文占用减少：三个平台的核心协作说明明显缩短，减少反复读取和复述流程所占的上下文。
- Codex 角色分层：`gpt56-role` 中 Explorer 使用 Terra/medium，Worker 使用 Sol/medium，设计与验收使用 Sol/high，少数深度验收使用 Sol/max，不再让所有角色继承同一高推理档位。
- 模型不可用时明确失败：显式固定的模型或推理档位不可用时会说明原因，不会悄悄换模型或降低档位。

`2.12.0` 没有独立发布；角色分层能力随 `2.13.0` 一起进入正式版本。

## 2.11.1 - 2026-07-08

小任务保持轻量：一行修复等轻量任务不自动增加问题、子任务或长期计划。

- 明确讨论要求优先：即使任务很小，只要你明确要求 `grill-me`、先讨论或先挑战假设，就会先遵从这个要求。
- 安装确认边界：安装和更新遇到尚未回答的关键问题会暂停，避免维护动作绕过你的确认。
- 维护者回归检查：检查会防止以后重新出现无意义提问、规划或过度分派；普通用户无需操作。

## 2.11.0 - 2026-07-08

关键决定先查证再询问：复杂或不确定任务会先调查，只把必须由你决定的问题带回来，并随问题给出推荐，减少把可自行查明的细节抛给你。

- 各阶段确认边界一致：Research、Debug、Plan、Execute、Review 和 Goal 不会出现一个阶段承诺等待、另一个阶段却继续执行的情况。
- 离线检查范围：检查会防止简单请求变成无意义访谈，但这些结果不代表真实平台运行已经得到证明。
- 安装内容漂移检查：`check-update.sh` 除了版本号，还能发现全局和当时项目级 skills、agents 的实际内容差异。
- 当时的配置选项：项目更新命令和 profile 示例改为可直接复制执行；新增 `gpt55-high`，可把 Codex 子任务固定为 GPT-5.5/high。

## 2.10.0 - 2026-07-08

维护者候选对比：旧版和候选版使用同一批任务，结果出来后再决定是否接受，不再只凭感觉判断提示词变化。

- 候选证据要求：每个候选都要留下模型、配置、验证结果、回滚方式和独立复查，不能只展示最好的一次输出。
- 失败方向留档：接受和拒绝的候选都会保留记录，避免以后重复尝试已经失败的方向。
- 普通用户无额外流程：这套工具只用于维护 Teamwork，不会给普通用户任务增加新阶段。
- 能力边界：本版本只提供可验证的优化基础设施，不代表 Teamwork 已经自动完成过一次真实优化。

## 2.9.0 - 2026-07-08

离线回归检查：维护者可用固定样例覆盖“小任务不过度流程化”、复杂编码、调试、调研、复查、长期目标和跨平台边界，更快发现规则或安装内容被意外改坏。

- 无模型调用：这些检查完全离线运行，普通用户无需承担额外调用或等待。
- 发布证据门槛：行为变化发布前需要开发证据、发布审计以及接受或拒绝记录，不能只因为“看起来不错”。
- 用户流程不受影响：评测只约束 Teamwork 的维护过程，不会变成普通用户任务的新阶段。
- 结果边界：离线检查只能证明规则符合记录的样例，不能证明 Codex、Cursor 或 Claude Code 的真实运行效果。

## 2.8.1 - 2026-07-08

先讨论时暂停相关动作：依赖决定的分析结论、方向选择、编辑和分派都会等待确认，不只是暂停写代码。

- 修改位置更准确：改代码前会先找到真正负责这项行为的现有路径，并确认如何验证，减少在错误位置另造一套实现。
- 复查复杂度：复查会检查没有证据支持的分支、默认值和备用路径，避免一次修复增加更多难维护路径。
- 更新内容检查：能发现本地仍在使用旧 skill、agent 或全局策略。
- 三平台边界一致：Codex、Cursor 和 Claude Code 使用相同的确认与代码维护边界。
